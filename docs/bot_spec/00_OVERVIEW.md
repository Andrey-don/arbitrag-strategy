# Архитектура бота МОЕХ — Обзор и Философия

## Цель проекта

Построить **production-ready торгового бота** для арбитража на МОЕХ.
Инструмент старта: пара TATN/TATNP (ТАТки). Расширение: фьюч-спот, золото, валюта.

Production = бот работает в реальных условиях без постоянного контроля человека.
Это значит: учтены все нештатные ситуации, есть failsafe на каждый сбой.

Бот в реальные деньги — только после полного прохождения курса.

## Двухслойная структура знаний

```
Слой 1 — Уроки (источник истины)
  docs/урок_NN_*.md          ← читать только при необходимости

Слой 2 — Инженерный (быстрый доступ при кодинге)
  docs/bot_spec/
  ├── 00_OVERVIEW.md         ← этот файл (читать первым при кодинге)
  ├── 01_STRATEGY_RULES.md   ← правила входа/выхода/объёма
  ├── 02_RISK_MANAGEMENT.md  ← лимиты, маржа, стопы
  ├── 03_PRODUCTION_EDGE_CASES.md  ← каталог нештатных ситуаций
  └── 04_INFRASTRUCTURE.md   ← API, брокер, стек, окружение
```

**Принцип чтения:** при работе над конкретным компонентом — читать только один
соответствующий файл из bot_spec. Уроки — только если нужен источник правила.

## Философия проекта

### 1. Production-first с первого дня

Каждый урок разбирается с вопросом: «Что это означает для бота в реальных условиях?»
Не «как это красиво сделать», а «что произойдёт, если это упадёт в 3 ночи».

### 2. Знания → компоненты (не уроки)

Уроки дают сырой материал. Bot_spec — дистиллят в инженерной форме.
Если правило есть в 01_STRATEGY_RULES.md, кодер не должен читать три урока.

### 3. Экономия токенов = скорость работы

Плоские файлы для агента — не grep по тексту.
Файлы в bot_spec — компактные, action-oriented, самодостаточные.
SESSION_BRIEF указывает что читать, bot_spec содержит ответ.

### 4. Obsidian — для мышления, bot_spec — для кодинга

Obsidian с [[wikilinks]] — инструмент для человека-исследователя.
Bot_spec — интерфейс между накопленными знаниями и агентом-кодером.
Это не конкуренция, это разные слои одного проекта.

### 5. Нет «временных решений» в production

Любой обход без обработки нештатной ситуации — технический долг.
Лучше не реализовать фичу, чем реализовать без failsafe.
