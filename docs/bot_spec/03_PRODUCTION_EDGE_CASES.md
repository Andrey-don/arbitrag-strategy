# Production Edge Cases — Каталог нештатных ситуаций

> Каждая нештатная ситуация = одно правило поведения бота.
> «Не знаю что делать» в production = катастрофа.
> Лучше консервативный failsafe (закрыть позицию), чем зависший бот.

## 1. CONNECTIVITY — Проблемы связи

### 1.1 Пропал интернет / выключился компьютер, позиции открыты
**Ситуация:** бот потерял соединение или компьютер выключился. Одна или обе ноги открыты.
**Риск:** позиция висит без мониторинга, спред может уйти в любую сторону, депозит под угрозой.

**Главная защита — выставлять Stop Orders сразу при входе:**
T-Invest API поддерживает `PostStopOrder` — условные ордера, которые хранятся
на серверах брокера и исполняются автоматически, даже если наш компьютер
выключен или нет интернета.

```python
# Сразу после исполнения обеих ног — выставить защитный стоп:
# stop_price = спред_входа + MAX_STOP_SPREAD (например +5 руб.)
# Для лонга TATNP + шорта TATN это означает:
#   при TATNP: stop_loss ниже цены входа
#   при TATN:  stop_loss выше цены входа

POST /tinkoff.public.invest.api.contract.v1.StopOrdersService/PostStopOrder
{
  "figi":            "...",
  "quantity":        N,
  "price":           стоп_цена,
  "stopPrice":       стоп_цена,
  "direction":       "STOP_ORDER_DIRECTION_SELL",  # или BUY для шорта
  "stopOrderType":   "STOP_ORDER_TYPE_STOP_LOSS",
  "expirationType":  "STOP_ORDER_EXPIRATION_TYPE_GOOD_TILL_CANCEL"
}
```

**Слои защиты (от надёжного к менее):**

| Слой | Механизм | Работает без интернета? |
|------|----------|------------------------|
| 1 (главный) | Stop Orders на сервере T-Invest | ✅ ДА |
| 2 | Watchdog: при восстановлении связи → проверить позиции | ❌ требует связь |
| 3 | Таймер принудительного закрытия за 30 мин до конца сессии | ❌ требует связь |

**Алгоритм Order Executor при входе:**
```
1. asyncio.gather(buy_leg, sell_leg)          # войти
2. asyncio.gather(post_stop_tatn, post_stop_tatnp)  # сразу стопы на сервер
3. Только после п.2 — считать позицию открытой
```

**При восстановлении связи (Watchdog):**
```
1. Запросить GetPositions → сравнить с локальным state
2. Запросить GetStopOrders → проверить что стопы живы
3. Если стопы исчезли (исполнились) → зафиксировать P&L, уведомить
4. Если позиция открыта > MAX_UNMONITORED_MINUTES без стопов → закрыть рынком
```

**Параметры:**
```python
MAX_STOP_SPREAD_RUB      = 5.0   # стоп при расширении спреда на +5 руб от входа
MAX_UNMONITORED_MINUTES  = 30    # мин без мониторинга → принудительное закрытие
WATCHDOG_INTERVAL_SEC    = 10    # как часто проверяем связь
```
**Статус:** РЕШЕНО (алгоритм) / TODO (имплементация)

## 2. EXECUTION — Проблемы исполнения

### 2.1 Одна нога исполнена, вторая — нет
**Ситуация:** нога 1 (например ЛОНГ TATN) заполнена. Нога 2 (ШОРТ TATNP) не заполнена.
**Риск:** открытая направленная позиция без хеджа.
**Алгоритм:** TODO
```
if leg1_filled and not leg2_filled:
    if time_since_leg1 < MAX_ONE_LEG_SECONDS:
        continue trying leg2
    else:
        # экстренное закрытие ноги 1
        close leg1 at market
        log "UNBALANCED_POSITION_EMERGENCY_CLOSE"
        alert operator
```
**Реальный кейс (урок_12):** автор закрылся в −15 рублей из-за ошибки с направлением.
**Статус:** TODO

### 2.2 Частичное исполнение (partial fill)
**Ситуация:** ордер исполнился частично (например, 250 из 1000 лотов).
**Алгоритм:** TODO

### 2.5 Ноги закрылись в разное время (асинхронный выход)
**Ситуация:** нога 1 закрылась по прокидке, нога 2 ещё открыта.
**Риск:** снова открытая направленная позиция.
**Алгоритм:** TODO
- После закрытия ноги 1: немедленно закрыть ногу 2 по рынку
- НЕ ждать «лучшей цены» для ноги 2 (риск > потенциальная прибыль)
**Реальный кейс (урок_14):** «закрыл одну ногу на прокидке, вторая ещё открыта — вторая может уйти против пока закрываешь первую»
**Статус:** TODO

## 8. PARTICIPANT — Работа от участника (р9-12, Каширин)

### 8.1 Участник исчез — позиция открыта
**Ситуация:** позиция открыта на основе айсберг-участника. Участник пропал (нет крупных заявок в стакане).
**Риск:** неэффективность закрылась или переместилась, спред не схлопнется.
**Алгоритм:** РЕШЕНО
```
1. Зафиксировать момент исчезновения участника
2. Наблюдать 2–3 рабочих дня: ждать вкат или возврат участника
3. Если через 3 дня участника нет и вката нет:
   → закрыться по рынку (убыток ≤ 2% от депо)
4. Если участник появился снова → можно перезайти
```
**Реальный кейс (р9-12):** «если нет участника абсолютно никого нет, объема нет — цена эффективная, закрываюсь в течение пары дней, не более 2%»
**Источник:** р9-12 (Каширин)
**Статус:** РЕШЕНО

```python
PARTICIPANT_WATCH_DAYS = 3        # дней ожидания после исчезновения участника
PARTICIPANT_EXIT_MAX_LOSS = 0.02  # 2% — максимальный убыток при таком закрытии
```

### 8.2 Вход в позицию перед выходом новостей
**Ситуация:** в стакане есть сигнал (раздвижка, айсберг), но через X минут выходит важная новость (открытие Америки, статистика ФРС, решение ЦБ).
**Риск:** новость резко двигает Форекс / ММВБ только в одну сторону → временный дисбаланс ног превышает допустимый стоп.
**Алгоритм:** РЕШЕНО
```
Если известна дата/время новости:
  → НЕ открывать новую позицию за 15 мин до новости
  → Если уже в позиции: добрать хедж до 80%
  → Сразу после выхода новости (она "отработала"): можно входить
  → При малом объёме: заходить сразу после новости с нормальным хеджем

Если позиция уже открыта во время новости:
  → Не паниковать, удерживать
  → Хедж 80% — защита от Форекс-выброса
```
**Реальный кейс (р9-12):** «в пятницу открытие Америки — валютный бакс улетел. Хорошая идея перед новостью не заходить. Кто системный и заботится о рисках — не заходит перед новостями.»
**Источник:** р9-12 (Каширин)
**Статус:** РЕШЕНО

```python
NEWS_BLACKOUT_MINUTES = 15     # минут до новости — не входить
NEWS_HEDGE_PCT        = 0.80   # хедж при позиции во время новости
```
